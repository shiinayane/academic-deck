Part I Reproduction objective

Reproduction question: is custom xArm RMPFlow worth keeping?

The work focused on whether the current xArm controller path can become a reliable main route for grasping and data collection.

  • Asset cleanupxArm7 is used with its original gripper, a TCP-centered end-effector frame, and the correct joint assumptions.
  • Controller testThe RMPFlow-style path was kept first, so the reproduction could tell us whether the current route is fixable.
  • Decision targetThe practical question is not whether RMPFlow can ever move the arm, but whether this custom path should remain the main grasping loop.
Reproduction framing
RMPFlow Finding Observed failure mode

The issue is not just orientation or a slow convergence problem

The best position-only runs still stabilize at a wrong equilibrium near 25 cm from the target.

Observed symptomPosition-only runs still converge to a stable wrong basin, with the best error around 0.249 m.
What did not fix itLonger horizons, orientation removal, default posture scans, damping, joint limits, and RMP parameter sweeps did not produce a robust route.
What was ruled outThe target is reachable by IK, and the URDF TCP position kinematics match the simulated robot.
InterpretationThe target is reachable and the task state machine is not the root cause. The failure sits in the custom RMPFlow/Lula configuration and its interaction with default posture and motion-policy parameters.
RMPFlow symptom
Decision Evidence Why not keep tuning

Why the current custom RMPFlow is not a good main route

The diagnosis ruled out several easy explanations and left us with a configuration basin that is hard to maintain.

  • Not the task FSMThe pick phases expose the issue, but the same target is geometrically reachable.
  • Not one bad parameterCollision spheres, default_q, damping, joint limits, target RMP, and axis RMP scans moved behavior locally but did not remove the main failure mode.
  • DecisionDo not keep investing in raw custom xArm RMPFlow as the main grasping loop. Keep geometry tools for support, but move manipulation toward learned policies.
RMPFlow decision
Controller Direction What replaces RMPFlow

Move the main route toward policy-based execution

The reproduction finding is not a failure of the project. It tells us where the engineering effort should go next: keep geometry tools for support, but make policy execution the main route for manipulation.

Working decisionUse IK / waypoint / Lula planner as controlled support paths, and prepare a new learned-policy path for robust skill execution.
  • KeepClean xArm asset assumptions: TCP frame, gripper drive joint, quaternion convention.
  • UseIK and waypoint planners for deterministic sanity checks and data generation support.
  • ShiftPrioritize ACT / Diffusion / remote OpenPI-style policies for manipulation skills.
  • MeasureSeparate planning validity from low-level skill execution success.
Policy direction
Part II Start from a plain request

The HRC idea is simple: let a human ask for lab help

Not "discover a new material" and not "replace the scientist". The first target is much smaller: a researcher gives a normal instruction, and the robot helps with physical lab work.

"Can you clean the beaker on the left and bring me two test tubes?"

A normal sentence for a human, but not a fixed LabUtopia task ID.
  • Human sideThe instruction is short, informal, and may be incomplete.
  • Robot sideThe system must turn it into concrete objects, skills, safety decisions, and execution logs.
Plain-language framing
Literature Takeaway Concise survey result

The idea is adjacent to active work, but the benchmark gap is still open

The survey result should be presented carefully: this is not a completely untouched area, but the exact LabUtopia-HRC package is not already solved.

Closest systemsORGANA and Yang et al. show natural-language chemistry robot assistants; RoboChemist shows long-horizon safety-compliant chemistry execution.
Closest benchmarksLABSHIELD / LabSafety Bench cover lab safety reasoning; PARTNR / AmbiK / Ask-to-Clarify cover HRC, ambiguity, and clarification.
Missing combinationNo public work clearly combines LabUtopia-style embodied lab simulation with natural-language HRC, ambiguity clarification, safety refusal, execution logs, and standardized baselines.
PositioningDo not claim "first lab robot assistant"; claim a reproducible benchmark and execution framework for lab HRC.
Related-work takeaway
Why It Is Hard What the robot must decide

A simple request hides several research problems

The key is not just language understanding. The robot has to decide whether it understands enough to act safely.

1
What is the user asking for?Clean, fetch, prepare, store, dispose, open, move, or operate a device.
2
Which physical object is meant?"The left beaker" may refer to multiple beakers depending on viewpoint and scene state.
3
Is it safe to execute?Unknown liquid, heat, contamination, fragile glassware, and device state can change the correct response.
4
Should the robot ask first?If the object or safety state is uncertain, asking is better than doing the wrong action.
5
Did execution actually work?The system needs monitoring and recovery, not only a generated plan.
HRC decomposition
Why Benchmark Not one number, a standard testbed

Here, "benchmark" means a repeatable way to test the whole loop

LabUtopia is already a benchmark because it provides tasks, environments, success conditions, and comparable results. LabUtopia-HRC would add human-language interaction as the thing being tested.

Fixed test casesSame scenes, same instructions, same object states, same ambiguous and unsafe cases.
Comparable systemsRule planner, LLM-only, scene-grounded LLM, SayCan-style planner, Code-as-Policies, and policy-based execution.
Multiple metricsTask success, object grounding, plan validity, clarification quality, unsafe rejection, recovery, and human turns.
Why it mattersIt tells us which part fails: language, grounding, safety, planning, low-level control, or recovery.
Benchmark framing
MVP 2-4 week target

Start narrow: typed instruction, fixed skills, measurable logs

The first version should be easy to explain and easy to run: no speech, no open-ended chemistry, no large VLA training dependency.
